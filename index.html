Agenda Item 2 · 10 minutes
Vendor Introduction — Microsoft & Dynamics 365
Agenda expectation: The vendor briefly introduces itself with a focus on (1) experience with similar levels of complexity, (2) where the solution typically creates the greatest value – UVP, and (3) key prerequisites or limitations important to understand from the outset. Not a presentation walkthrough — set the scene for the dialogue.
Experience with Similar Complexity
Microsoft Dynamics 365 is used by member-based and employer organisations across Europe. Comparable reference customers include Confederation of Norwegian Enterprise (NHO), Confederation of Danish Employers (DA), and several Scandinavian trade associations managing 10,000–50,000+ member companies. The common denominators: complex member hierarchies, multi-level committee structures, 500–2,000+ daily CRM users, and a drive to consolidate fragmented legacy systems onto a single, trusted data foundation.
Unique Value Proposition for DI
D365's strongest differentiator for DI is the Microsoft ecosystem coherence: Teams, Outlook, SharePoint, Power Platform, and Azure are already in DI's environment. D365 turns these from isolated tools into a unified member engagement platform — phone calls logged automatically, emails tracked in one click, meetings prepared by Copilot in 30 seconds. No integration middleware needed for Microsoft-to-Microsoft connections. This is where DI's 1,000 daily users will feel the difference immediately.
Where D365 Creates the Greatest Value for DI
🔗
Microsoft Ecosystem Integration
Native Teams, Outlook, and SharePoint integration eliminates manual switching between tools. Already in DI's tenant — zero integration cost for these connections.
⚙️
Self-Service Configurability
Power Platform gives DI's own team the ability to adapt the data model, workflows, and forms without vendor dependency — escaping the MemberCare customisation trap.
🤖
Copilot — AI That Works Today
Copilot for Sales is in production today — not a roadmap item. Meeting summaries, account briefings, and next-action suggestions are live features, not future promises.
📊
Dataverse as Integration Hub
Replacing MemberCare's 32-system point-to-point integrations with a standards-based API and event architecture. Dataverse becomes DI's system of record.
🔄
D365 F&O Already in House — Dual-write Advantage
Since DI already runs D365 Finance & Operations, the CRM-to-ERP integration is not a custom project — it is a native, included feature called Dual-write. Real-time bidirectional sync between CE and F&O is out of the box. Membership fee invoicing, payment status, and customer master sync are covered without building anything.
Key Prerequisites & Limitations — Stated Upfront
  • No native membership fee engine. D365 does not ship with purpose-built kontingent/subscription billing. This requires configuration of a custom fee-rule model in Dataverse + Power Automate. However: since DI already uses D365 F&O, the invoicing side is fully covered — Dual-write propagates the calculated fee to F&O where invoice creation, AR, and payment collection happen natively. This is not a custom integration — it is included.
  • Complexity of the data migration from MemberCare. 14 years of member history, 32 existing integrations, and custom MemberCare data structures require a dedicated data migration workstream. This is the single largest project risk and must be scoped carefully in the RFI.
  • Adoption investment is real. With 1,000 daily users across 7 different roles, change management is not optional — it is a critical workstream. D365 is more capable than SuperOffice, but more complex to roll out at scale without dedicated training and change support.
  • Conversation Intelligence requires Teams Phone. Automatic phone call transcription and AI summarisation requires Microsoft Teams Phone as the telephony layer. If DI uses a different telephony provider, this capability requires a custom connector or may not be available in full.
  • Event module is functional but not best-of-breed for DI's 2,000-event scale. Customer Insights – Journeys includes event management, but for large-volume, complex ticketing scenarios (e.g., paid events with multiple ticket tiers, waiting lists, complex capacity logic), a specialist event platform (like Cvent or Eventbrite) with a Dataverse integration may deliver a better experience than the native module alone.

    ConfigurePower Automate

    ConfigureDashboards

Agenda Item 6 · 20 minutes
Vendor Perspective & Challenge Session
Agenda expectation: DI wants genuine market insight and honest input that can challenge current assumptions and calibrate ambition. The session explicitly asks for what we would challenge, what the risks are, where the solution is less suited, what typical mistakes organisations like DI make, and which integration/architecture decisions are most critical to clarify ahead of the RFI. This is not a Q&A session — it is a consultative challenge.
1
What would you challenge in DI's approach or setup to increase the chances of success?
▼
We would challenge three assumptions directly:
  • Assumption: replacing the system solves the data quality problem. DI's pre-analysis cites fragmented data models and synchronisation delays as core issues. But these are partly organisational — if the 1,000 users don't adopt the new system consistently, bad data will re-emerge in any platform. A D365 implementation without a funded change management and data governance programme will reproduce the same problems in a more expensive wrapper. Our challenge: Where is the plan for people and process change, not just technology change?
  • Assumption: all 32 MemberCare integrations need to be rebuilt. A phased integration strategy — where only mission-critical integrations go live on day one — is almost always safer than a big-bang migration. Our challenge: Is DI prepared to accept a period of reduced integration coverage in exchange for a faster, lower-risk go-live?
  • Assumption: the RFI will be vendor-neutral. If DI is already heavily invested in Microsoft 365, Teams, and Azure, a non-Microsoft CRM will require significant integration effort for what D365 delivers natively. We would challenge DI to explicitly quantify the integration cost premium of a non-Microsoft choice before the RFI concludes.
2
What are the biggest risks or pitfalls in a project like this?
▼
Based on comparable membership-organisation CRM programmes, the three most common failure modes are:
  • Data migration underestimated. Moving 14 years of MemberCare data — with its custom schema, custom integrations, and accumulated data quality issues — is consistently the activity that consumes the most budget and causes the most delays. Projects that treat data migration as a technical task rather than a business-led programme almost always overrun. Mitigation: commission a data assessment before the RFI, not after.
  • User adoption failure. D365 is a capable but complex platform. A 1,000-user rollout across 7 user groups with different workflows, roles, and data needs is a significant change programme. Organisations that skimp on training, role-based configuration, and post-go-live support see adoption plateau at 40–60%. Mitigation: budget change management as 15–20% of the total implementation cost from day one.
  • Scope creep driven by configurability. The Power Platform's low-code extensibility is a strength — but it can become a liability when every business unit wants to add their own customisations during implementation. Without a strong governance model, the platform accrues technical debt faster than MemberCare did. Mitigation: appoint a Power Platform governance owner in the DI team before go-live.
3
Where is Dynamics 365 particularly strong — and where is it less well suited to DI's needs?
▼
Honest assessment:
  • Particularly strong: Microsoft ecosystem coherence (Teams + Outlook + SharePoint = zero integration cost); Power Platform self-service configurability; Copilot for Sales (genuinely production-grade today); Dataverse as a unified data platform for all of DI's data; enterprise security, compliance, and GDPR tooling; scalability to DI's user volumes without per-seat performance constraints.
  • Less well suited: Native membership/association-specific features. D365 Sales is built for B2B commercial sales organisations. Concepts like membership tiers, quarterly fee calculation, committee governance, and annual event programmes are not native — they require configuration. DI's procurement team should evaluate whether the configuration effort is worth the platform's other strengths, compared to a purpose-built membership management system (e.g., iMIS, MemberSuite, Salesforce for Nonprofits).
  • Less well suited: Deep event management at scale. 2,000+ annual events with paid ticketing, complex capacity models, and multi-channel registration workflows push beyond what Customer Insights – Journeys was designed for. A specialist event platform integrated via API is likely the better long-term answer for DI's event volume.
  • Maturity gap: Some Copilot features are still maturing. Copilot in Customer Insights – Journeys (real-time personalisation at scale) is less mature than Copilot in D365 Sales. DI should validate specific AI features in a sandbox environment before committing.
4
What typical mistakes do you see organisations like DI make in projects like this?
▼
Five patterns we see repeatedly in membership-organisation CRM programmes:
  • Trying to replicate MemberCare in the new system. The temptation is to map every existing field, every existing workflow, and every existing screen to the new platform. This produces a D365 that behaves like a slow, expensive version of the old system. The right approach: redesign processes around the new platform's strengths, not the old system's limitations.
  • Treating the CRM project as an IT project. The most successful CRM implementations are business-led, with IT in a delivery support role. DI's core project team includes CRM & Loyalty leads — that is the right structure. The risk is if business ownership weakens during implementation when day-to-day demands compete with project work.
  • Selecting too many modules on day one. D365 Sales + Customer Insights – Journeys + Customer Service + Power Platform + Copilot, all in scope from day one, creates an implementation programme that is very hard to deliver within time and budget. Recommendation: define a clear MVP scope for the initial go-live and a post-go-live enhancement roadmap.
  • Underestimating the Dataverse data model design phase. Getting the entity relationships right — especially DI's committee and board position model — before build begins is critical. Retrofitting a data model after data migration has started is expensive. This phase deserves 4–6 weeks of focused workshops before any configuration begins.
  • No dedicated internal system owner post go-live. DI will accumulate configuration changes over time. Without a dedicated Power Platform administrator who owns the platform, governs changes, and maintains documentation, the system will drift back toward MemberCare's state of "nobody really understands it" within 18–24 months.
5
Which integration or architectural decisions are most critical to clarify ahead of the RFI?
▼
Five architectural decisions that will shape the entire project:
  • Telephony platform decision. Conversation Intelligence and automatic call logging require Teams Phone (or a certified CCaaS integration). If DI uses a different telephony provider, this must be resolved — or the phone-automation capability of UC2 must be descoped. This decision should be made before the RFI, not after vendor selection.
  • Integration hub architecture. Will DI adopt Dataverse as the system of record (all systems read/write via Dataverse APIs), or maintain a separate integration bus? The former is simpler and more cost-effective for a Microsoft-stack organisation; the latter may be required if DI's existing Azure environment already has a mature API management layer. Clarify this before writing RFI integration requirements.
  • Finance system connection. Since DI already uses D365 F&O, this decision is largely made — Dual-write is the native integration between CE and F&O and is included with qualifying D365 licences. The question to clarify is not which ERP, but which Dual-write entity maps are activated: Account↔Customer, Contact↔Contact, Sales Order↔F&O Sales Order, and — most critically for DI — how the membership fee calculation output triggers an F&O free-text invoice or sales order. This flow should be designed in Phase 0 of the implementation.
  • Min Side (member portal) rebuild vs. integrate. The current Min Side is a self-built portal. Should it be rebuilt on Power Pages (part of the Power Platform, natively connected to Dataverse), or maintained as a separate application integrated via API? Power Pages reduces integration complexity but requires UX design work. This decision shapes the integration scope significantly.
  • Event platform decision. Is the event module in scope for the CRM replacement, or managed separately? If in scope, decide upfront whether to use Customer Insights – Journeys events (configurable, native) or a specialist platform (Cvent, Eventbrite) integrated via Dataverse. This determines one of the largest integration workstreams in the project.

Commercial Framework — Pricing Model, Implementation & Timeline
Agenda expectation: Overview of the pricing model and key cost drivers, implementation approach, and typical timelines. Not detailed pricing — a framework for understanding the commercial structure.
📄
Licensing Model
D365 is per-user, per-month subscription licensing. DI's 1,000 users would be split across license tiers:
  • D365 Sales Premium — full CRM + Copilot for Sales. For relationship managers, sales/recruitment (~200–300 users)
  • D365 Sales Enterprise — core CRM. For member service, admin users (~400–500 users)
  • Power Apps per-user — for light users needing form access only (e.g., committee portal users)
  • Customer Insights – Journeys — tenant-level pricing (not per user), based on marketed contacts
Microsoft EA customers (which Microsoft is) typically receive significant discounts vs. list price. Exact pricing provided in RFI response.
⚙️
Key Cost Drivers
The total cost of ownership has four main components:
  • Licences — recurring annual subscription, negotiated via EA
  • Implementation — data model design, configuration, data migration, integration development, and training. Typically the largest one-time cost
  • Data migration — often under-budgeted; 14 years of MemberCare data requires dedicated workstream
  • Change management & training — 1,000 users across 7 roles; budget 15–20% of implementation cost
  • Post go-live support & enhancements — ongoing Power Platform admin, feature releases, and evolution
🚀
Implementation Approach
Microsoft recommends a phased approach for DI's scale:
  • Phase 0 (8–12 wks) — Data model design, data assessment, architecture decisions, environment setup
  • Phase 1 MVP (20–28 wks) — Core data model, UC1 + UC2 scenarios, Outlook/Teams integration, basic dashboards, pilot with 50–100 users
  • Phase 2 (16–20 wks) — Member dialogue / marketing journeys, event integration, remaining user groups, full rollout
  • Phase 3 (ongoing) — Fee logic, advanced AI features, Power Platform self-service enhancements
Microsoft FastTrack (included with qualifying D365 licences) provides remote architecture and technical guidance throughout.
📅
Typical Timeline
For a comparable membership-organisation CRM programme (1,000 users, complex member hierarchy, legacy migration):
  • Phase 0 start → MVP go-live: 8–10 months
  • MVP → full rollout: additional 4–6 months
  • Total: 12–18 months from project start to full organisational rollout
Key milestone: DI's RFI invitation is expected late June 2026, with responses in August 2026. A contract signed in Q4 2026 → MVP go-live Q3 2027 is a realistic target.
Note on detailed pricing: Indicative total cost of ownership for DI's profile (1,000 users, full implementation, Customer Insights – Journeys, Power Platform) will be provided in the formal RFI response. Microsoft's pricing is structured to be transparent and predictable — no usage-based surprises for CRM core functionality. AI features (Copilot) are included in Sales Premium; Customer Insights is priced separately per marketed contacts/unified profiles.
🤝
Delivery Model
Microsoft Partner Ecosystem
D365 implementations are delivered by certified Microsoft Partners — not directly by Microsoft. DI should evaluate partner options based on: (1) experience with membership-organisation CRM implementations in Denmark/Scandinavia, (2) size and capability to handle DI's 1,000-user scale, and (3) long-term support model. Microsoft's FastTrack for Dynamics 365 programme provides Microsoft-direct technical validation at no additional cost for qualifying deals — a built-in quality assurance layer on the implementation.
